## 概述

将本次会话产生的所有知识持久化：工程经验、功能状态、已修复 Issue、创意和记忆更新。

**开始时宣告：** "正在保存项目资产…"

---

### 触发条件

**✅ 主动触发（无需用户明确要求）：**
| 信号 | 示例 |
|------|------|
| 用户确认某功能可用 | "可以了"、"验证通过"、"已修复"、"it works" |
| 会话即将结束 / 上下文接近上限 | `/compact` 前、长时间停顿前 |
| 功能分支合并到 main | `git merge` 完成或 PR 合并后 |

**✅ 用户明确要求时触发：**
| 说法 | 执行范围 |
|------|---------|
| "保存资产" / "save assets" | 完整 7 步 |
| "更新文档" / "update docs" | 仅第 2-3-7 步 |
| "更新记忆" / "update memory" | 仅第 6 步 |
| `/save-project-assets-zh` | 直接调用 |

**❌ 不触发的情况：**
- 每次 `git commit` / `git push` — 太频繁，第 7 步本身会产生 commit
- 仅说"保存"/"记录"而无明确意图 — 太模糊，可能是保存文件/记录 todo
- 测试失败或功能仍在进行中 — 尚无值得保存的内容
- 小型重构 / 格式修复提交 — 无新知识产生

---

**会话开始：** 静默加载 MEMORY.md 以恢复上下文，除非用户询问否则不宣告。

---

## 第零步：内容检查（快速判断）

执行任何操作前，先回答：

- 本次会话是否产生了**新的**工程经验、修复或架构决策？
- 用户是否**明确确认**了某个功能可用？
- 是否有**新的**想法或状态变更尚未写入文档/记忆？

**若全部为否 → 立即停止。** 告知："本次会话无新内容需要保存。"

至少一个为是时，才继续执行后续步骤。

---

## 第一步：采集本次会话知识

写入任何内容之前，先整理本次会话产生了什么：

| 分类 | 来源 | 目标文件 |
|------|------|---------|
| 工程经验 | 已修复 Bug、发现的根因、架构决策 | `docs/TECH_LOG.md` |
| 功能状态 | 已构建/测试/合并的内容 | `docs/CHANGELOG.md` |
| 已修复 Issue | 用户确认可用的 Issue | GitHub Issue 评论 |
| 创意 / 待办 | 发现但未实现的改进点 | `docs/IDEAS.md` |
| 持久事实 | 架构、偏好、约束 | 记忆文件 |

**只写新内容** — 不重复已有条目。

---

## 第二步：更新 TECH_LOG.md（经验教训）

文件：`docs/TECH_LOG.md`

在**顶部**插入（最新优先）：

```markdown
## {YYYY-MM-DD}：{简短标题}（Issue #{N}）

### 问题
{出了什么问题 / 面临什么挑战}

### 根因
{为什么会发生——具体指出代码/API/行为}

### 修复方案
{做了什么；若有参考价值可附关键代码片段}

### 经验总结
- **规则**：{未来可执行的行动准则}
- **原因**：{为什么这很重要}
```

**本次会话无新经验则跳过。**

---

## 第三步：更新 CHANGELOG.md（功能状态）

文件：`docs/CHANGELOG.md`

- 有新功能/修复完成 → 在顶部新增或更新 `[version-dev]` 章节
- 格式：`### 🐛 修复` · `### 🚀 新功能` · `### 📐 重构`
- 包含 Issue 编号、关键行为变更、测试数量
- **不重复**已有条目

---

## 第四步：评论已修复的 GitHub Issue

对本次会话用户确认修复的每个 Issue：

```bash
gh issue comment {N} --repo {YOUR_GITHUB_REPO} --body "## ✅ 已修复

{2-3 句话：根因 + 修复方案 + 验证方式}

Status: **fixed**"
```

**只有用户明确确认可用时才评论。** 不推测。

---

## 第五步：更新 IDEAS.md（未来工作）

文件：`docs/IDEAS.md`

本次有新想法或"发现但未做"的内容：

```markdown
## {功能方向}

- [ ] {具体想法} — {为什么值得做}
```

**无新内容则跳过。**

---

## 第六步：更新持久记忆

记忆目录：Claude Code 自动位于 `~/.claude/projects/{sanitized-cwd}/memory/`

### 6a. 更新 MEMORY.md 索引
- 为新记忆文件添加索引指针
- 若事实已变更则更新现有条目（新测试数量、新分支、新版本等）

### 6b. 按需写入/更新记忆文件

**何时新建记忆文件：**
- 发现新的架构模式
- 具有未来参考价值的新工程经验
- 项目状态变更（新分支、功能上线、新 Issue 模式）
- 用户偏好或工作流纠正

**记忆文件格式：**
```markdown
---
name: {主题}
description: {一行描述，用于未来相关性匹配}
type: project | feedback | user | reference
---

{内容}
**原因：** {动机}
**如何应用：** {何时需要注意}
```

### 6c. 需保持最新的关键记忆
- 当前活跃分支及其用途
- 最新测试数量
- 最新发布版本/产物名称
- 已修复 Issue 列表（追加新修复项）
- 新工程经验

---

## 第七步：提交文档

若 `docs/` 文件有修改：

```bash
git add docs/TECH_LOG.md docs/CHANGELOG.md docs/IDEAS.md
git commit -m "docs: {简短描述本次记录内容}"
git push origin {当前分支}
```

---

## 会话开始协议

新会话开始时：

1. Claude Code 自动加载 MEMORY.md — 静默吸收其中事实
2. 记录：活跃分支、开放 Issue、最近经验、最新测试数量
3. 无需重读源码即可回答项目状态问题
4. 若用户首条消息涉及近期工作，确认上下文："我有上次会话的上下文：{关键事实}。"
